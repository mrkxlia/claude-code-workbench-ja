# タスクパイプライン ハードルール（常に適用）

（原本 `task-pipeline/CLAUDE.md` の「ハードルール」を Kiro steering 化したもの。ツール非依存のルールのみ。）

1. **人間承認チェックポイントを飛ばさない** — 要件承認・ブリーフ承認・最終レビューの3点では、明示承認があるまで先に進まない。
2. **出力ディレクトリの外に書かない** — 成果物ビルダーが作成・編集してよいのは出力ディレクトリだけ。コード・既存資料・設定には触れない。
3. **担当領域を越境しない** — レビュアーは成果物を直さない。ビルダーは要件・ブリーフを書き換えない。
4. **要件を推測で埋めない** — 不明点は「未解決の質問」として人間に確認する。
5. **機密情報・個人情報・トークンを成果物に書き込まない。**
6. **受け入れ基準のセルフチェックをせずに「完了」と言わない。**
7. **既存仕様（SPEC.md・成果物仕様）に黙って反しない** — `[確定]` 要件（`D-NN`）を変える/廃止するならブリーフの「既存仕様への影響」で明示し承認を得る。
   承認された変更は最終承認後に SPEC.md へ増分反映する（原則「1 Todo = 1 Commit = 1 Spec Update」）。

## 表記・スタイル規約（プロジェクトに合わせて差し替え）

- ドキュメントは日本語で書く。見出しは体言止めに統一する。
- 英数字は半角。日本語と英数字の間にスペースは入れない。
- 図の凡例は右下に置き、配色は既存の図に合わせる。
- 日付は `YYYY-MM-DD` 形式で書く。

## 成果物の種類と出力先（プロジェクトに合わせて差し替え）

| 成果物 | 形式 | 出力先 |
|---|---|---|
| アーキテクチャ図・フロー図 | `.drawio` | `deliverables/diagrams/` |
| 設計ドキュメント・手順書 | `.md` | `deliverables/docs/` |
| 調査レポート | `.md` | `deliverables/reports/` |
