# ソフトウェアパイプライン ハードルール（常に適用）

（原本 `software-pipeline/CLAUDE.md` の「ハードルール」を Kiro steering 化したもの。ツール非依存のルールのみ。）

1. **機密ファイルは絶対にコミットしない** — `.env`・`*.key`・`*.pem`・`secrets.json` をステージしない。
2. **担当領域を越境しない** — バックエンド担当はフロントエンドのファイルに触れない。逆も同じ。テスト担当はプロダクトコードを直さない。
3. **人間承認チェックポイントを飛ばさない** — ストーリー承認・ブリーフ承認・最終レビューの3点では、明示承認があるまで先に進まない。
4. **新しい依存パッケージを勝手に追加しない** — 必要なら提案して承認を待つ。
5. **テストと型チェックを走らせずに「完了」と言わない。**
6. **ビジネスルールを推測で埋めない** — 不明点は「未解決の質問」として人間に確認する。
7. **既存仕様（SPEC.md）に黙って反しない** — `[確定]` 要件を変える/廃止するならブリーフの「既存仕様への影響」で明示し承認を得る。
   承認された変更は Phase 7 で SPEC.md に増分反映する（原則「1 Todo = 1 Commit = 1 Spec Update」）。

## アーキテクチャ原則（プロジェクトに合わせて差し替え）

- ビジネスロジックはサービス層に置き、API ルートは薄く保つ。
- すべてのクエリはテナント（組織）ID でスコープする。テナント分離をすり抜けるクエリを書かない。
- 日時は UTC で保存し、表示時にユーザーのタイムゾーンへ変換する。
- 生の決済ペイロード・個人情報・トークンをログに出さない。生のエラーをクライアントに返さない。
- `any` で型エラーを黙らせない。マイグレーションは後から手で書き換えず新規を足す。
