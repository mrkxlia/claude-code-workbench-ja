# clarify — 要件・構成を一問ずつ詰める徹底質問スキル

成果物の要件・構成・方針のあらゆる側面について、共通の理解に達するまでユーザーに
徹底的に質問する。作成を始める前に、穴・曖昧さ・隠れた前提を潰すのが目的。

> このスキルは外部の2スキルの設計を参考にしている:
> [dig](https://github.com/ryonakae/dotfiles/tree/master/config/.agents/skills/dig)（ryonakae/dotfiles）と
> [grill-me / grilling](https://github.com/mattpocock/skills)（mattpocock/skills）。
> ※ プロトコルの骨子は software-factory 版の clarify と同一で、語彙だけ成果物向けに調整している。

## 中核プロトコル

1. **質問は一度に一つ。** 回答を待ってから次へ進む。一度に複数質問を投げると相手は混乱する。
2. **各質問の前に調べる。** 調査レポート・既存の成果物・素材・規約を先に確認し、
   **調べればわかることは聞かない。** 質問は「人間にしか答えられないこと」に絞る。
3. **深さ優先。** 「幅」より「深さ」。1つのテーマを掘り尽くしてから次のテーマへ進む。
   構成の各分岐をたどり、判断の依存関係を一つずつ解決する。
4. **各質問に推奨回答を添える。** なぜその推奨かの理由も書く。甘い回答・曖昧な返答には突っ込む。
5. **AskUserQuestion ツールを使う場合は、`questions` 配列に常に1問だけ入れる。**
   ツールが複数質問を載せられても束ねない（中核プロトコル1「一度に一つ」を厳守）。
   選択肢には推奨を先頭に置き、ラベル末尾に「（推奨）」を付ける。

## 質問フォーマット（チャットで提示する場合）

```
### Q[番号]: [質問文]

[なぜこの質問が重要か — 1〜2行]

- **A** — [選択肢]
- **B** — [選択肢]
- ...

**推奨: [A/B/...]** — [理由]
```

AskUserQuestion ツールで提示する場合は、同じ内容を `question`／`options`（推奨を先頭・ラベルに「（推奨）」）／
各 option の `description` に対応させる。いずれの場合も**1問ずつ**。

## 終了

- 論点を掘り尽くし、新たな洞察が出なくなったら終了を提案する。ユーザーが続行を求めれば続ける。
- 終了時は「決まったこと」をまとめる:

```
## まとめ

### 決まったこと
- [論点]: [決定] — [根拠/前提]
- ...

### 残った未解決の質問（あれば）
- ...
```

## やらないこと

- 一度に複数の質問を投げる（混乱の元）
- 調べればわかることを質問する（先に調査する）
- 推奨回答を省く（ユーザーの判断を助けるために必ず添える）
- 曖昧な回答を曖昧なまま受け入れる（突っ込んで具体化する）
- 自分で決めてよい瑣末事まで質問する（重要な判断・依存関係に絞る）

## task-factory との連携

task-factory の Phase 2（要件）・Phase 3（ブリーフ）では、**writer（requirements-writer /
brief-writer）を起動する前に**工場長がこのプロトコルで要件・構成を詰める。
「決まったこと」を writer への入力に織り込み、その上で writer → 提示 → 🛑承認 と進む。
承認🛑は従来位置のまま（承認のたびに clarify を再起動しない）。作成フェーズ（Phase 4 以降）には
立ち入らない — 要件・構成の確定までが clarify の担当範囲。
