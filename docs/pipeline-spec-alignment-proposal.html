― 既存仕様の「吸い出し」と以降要件の「合致性」を強制化する

エグゼクティブサマリ（30秒）

  • 課題: 両パイプラインは「概要の機能・タスク要件」はブレずに流せるが、対象リポジトリの既存仕様・既存内容との整合性が保証されない。
  • 要求（ユーザー）: ①既存リポジトリからの仕様の吸い出し（extraction）を必須化、②以降の要件・成果物がその仕様に合致すること（conformance）を必須化。
  • 研究結論: 仕様駆動開発（SDD）の業界標準は「永続的で“生きた”仕様・規約の層」を要件→設計→実装の下に敷くこと（Kiro steering / Spec Kit constitution）。最大の敵はコンテキスト・ドリフト（一度作った仕様が放置で陳腐化）。
  • 推奨: 案A 軽量強化 — 新ファイル規約を増やさず、既存の SPEC.md / researcher / validator / spec-extract を強化し、吸い出しと合致を「任意」から「必須」へ格上げ＋SPEC.md を“生きた仕様”として維持して drift を断つ。
  • 対案: 案B steering 層新設 — Kiro 流に docs/steering/ を新設。最強だが規約層自体が drift し、メンテ・トークン・導入摩擦が増える。
  • 決定: 案A 採用。さらにユーザーの追加要望5件を案Aに反映（§7 M1〜M5）— 小変更の軽量追従／空リポ・既存リポの clarify ヒアリング／要件変更の改訂管理／既存からの再構成は read だけでなく clarify でとことんヒアリング。

1. 背景と要求

task-pipeline・software-pipeline は、依頼された機能・タスクの概要要件をブレずに流すことはできる。 一方で、対象リポジトリにすでに存在する仕様・規約・実装内容との整合性が保たれるかは保証が弱い、というのが本件の出発点である。

ユーザーが求めるのは、次の2点の強制化（enforce）:

  1. 吸い出し（extraction） — 既存リポジトリから要件・仕様を取り出す工程を必須にする。
  2. 合致（conformance） — 以降に作る要件・成果物が、その吸い出した仕様に合致することを必須にする。

本ドキュメントは「軽量強化（案A）」と「steering 層新設（案B）」を比較する判断材料。実装はユーザーの案選択後に着手する。

3. 既存パイプラインが既に持つ整合性機構（再発明しない）

本リポジトリのパイプラインは、すでに整合性機構をかなり持っている。これらは活かす。

  • codebase-researcher / source-researcher — 作る前に既存コード・パターンを file:line 付きで地図化。
  • spec-writer / builder — 「既存パターンの再利用を最優先」「CLAUDE.md と矛盾する設計を書かない」が明文ルール（spec-writer.md）。
  • implementation-validator — 項目5で「CLAUDE.md や既存コードと食い違うパターン」を検査（implementation-validator.md:37）。
  • spec-extract — レガシーコードの挙動を SPEC.md に逆引きし、全記述に [確定]/[推定]/[不明] の確度ラベル。
  • LEARNINGS.md → CLAUDE.md 昇格 / pipeline-improve — 運用からルールを育てる自己改善ループ。

案A — 既存プリミティブの強化 推奨・軽量

新しいファイル規約は増やさず、既存の SPEC.md / CLAUDE.md / researcher / validator / spec-extract / pipeline-improve を強化し、吸い出しと合致を「任意」から「必須」へ格上げする。

  1. 吸い出しの強制化（入口） — SPEC.md が無いレガシーは spec-extract を前工程として必須化（現状は README で「任意」）。researcher に SPEC.md＋規約ドキュメントの一次ロードを義務化し、依頼が既存仕様と矛盾したら「未解決の質問」で止める。
  2. SPEC.md を“生きた仕様”に（drift を断つ） — feature-pipeline の Phase 7（最終承認後）に、確定した新挙動を SPEC.md へ反映する増分更新ステップを追加。spec-extract の確度ラベルを流用。「入口=全体逆引き／出口=該当要件の増分更新」へ。
  3. 合致ゲートの明示化（強制） — validator / reviewer に「SPEC.md・既存仕様との整合性」を独立チェック項目として立て、SPEC.md を名前付き入力にし、矛盾を Critical 扱い。spec-writer は brief×SPEC を突合し仕様変更を明示列挙（沈黙の drift を禁止）。
  4. task-pipeline の前工程を新設 — source-researcher に既存規約/仕様/テンプレの一次ロードを義務化＋規約キャプチャ前工程を追加。brief-writer・deliverable-reviewer に整合ゲートを追加。
  5. implementation-skills 原本の更新 — spec-extract 原本に「増分更新／生きた仕様」運用を追記、notes 原本に「逸脱は SPEC.md 更新候補」を追記。連携版コピーを同期。

トレードオフ: 低複雑・低メンテ。リポジトリのミニマリズムと Anthropic の progressive-disclosure に整合。全プリミティブを再利用。 残リスク① 大規模コードで SPEC.md が肥大 → spec-extract 既存の「1モジュールずつ」スケーリング規則で緩和。 残リスク② 「義務化」はプロンプト指示ベースのため、案B の構造的な必須ロード保証より徹底度が下がりうる（＝穴②が完全には消えない）→ validator の独立 SPEC 整合ゲート（合致を後工程で構造的に二重化）で緩和。

案B — steering 層を新設 本格・大きい

Kiro/Spec Kit に倣い docs/steering/（product.md/structure.md/tech.md/testing-standards.md）の構造化規約層を新設し、全エージェントに必須ロード。CLAUDE.md は hot constitution としてそれを指す。案Aの内容も全部含む。

トレードオフ: 最強・最も明示的で Kiro/Spec Kit と 1:1。ただし (i) 複数ファイルの規約層自体が drift する（まさに Bootstrapping Illusion）、 (ii) エージェントごとのロードトークン増、 (iii) リポジトリの「単一 100–300 行 CLAUDE.md」方針と衝突、 (iv) テンプレ表面積・利用者のカスタム負担・導入摩擦が増える。

6. 推奨と理由

案A を推奨する。

  • ユーザー要求（吸い出し必須化＋以降の合致必須化）を、既存プリミティブで過不足なく満たせる。
  • 「SPEC.md を生きた仕様として維持」する 1 手で、研究が指摘したドリフトのループを閉じる。これが整合性の本丸。
  • Anthropic のトークン指針（段階ロード・500 行ルール）とリポジトリのミニマリズムに沿う。
  • 案B の追加構造は、CLAUDE.md＋生きた SPEC.md で実質カバー済みの内容を、より高い drift/メンテ費用で再実装する面が大きい。
折衷案: 案B の“語彙”だけ採用する。維持された SPEC.md を「spec of record / steering baseline」と呼び、概念的には Kiro steering と同じ役割だと位置づけつつ、ファイルの増殖は避ける。費用対効果が最も高い。

仕組み M1〜M5 の要約

M1 — 生きた仕様の二経路更新 #1・#4

大きな変更は feature-pipeline Phase 7 で SPEC.md を増分更新。小さな/スキル外の変更は notes 拡張で該当 F-NN だけ軽量更新（フル pipeline 不要）。変更・廃止は F-NN を維持しつつ supersede＋改訂履歴。やさしい注意喚起: ソースが変わって SPEC.md 未更新なら非ブロッキング通知（作業は止めない）。

既存設計を壊さない: notes は元々「逸脱は spec-extract の証拠源」（notes/SKILL.md の Relationship to spec extraction 節）という位置づけを持つため、M1 はその自然な延長。notes の「Signal, not a changelog」方針は維持し、SPEC 同期は該当 F-NN への増分だけに限定して changelog 化させない。

M2 — clarify 連結の仕様再構成 #5

spec-extract は「読むだけ」で終わらせない。草案後に [不明]/[推定] を clarify プロトコルで一問ずつ・推奨回答つきでヒアリングし、ユーザーの意図を引き出して [確定] に格上げする。やさしく・とことん・取りこぼさない。

M3 — 初期化＋既存リポ確認 #2

空リポ: clarify で種別（ソフト開発／ドキュメント等の成果物／その他）と初期 SPEC/要件を前向きに作成する。既存リポ: 自動検出だけで進めず、生成前に clarify 確認を挟む。新スキルは作らず既存 setup の分岐＋相互リンクで実現。

入口と振り分けの受け皿: 入口はユーザーが叩いた setup（/pipeline-setup または /task-pipeline-setup）。冒頭の種別ヒアリングでその setup の守備範囲と食い違ったら、他方の setup へ相互リンクで誘導する（コード→software-pipeline、図/文書等の成果物→task-pipeline）。「その他」は汎用の task-pipeline を既定の受け皿とし、どうしても当てはまらなければ clarify で最寄りの形を一緒に決める。

M4 — 必須ロード＋合致ゲート 案A 中核

researcher が SPEC.md を一次ロード／writer は矛盾を「未解決の質問」へ／spec-writer・brief-writer は仕様変更を明示列挙／validator・reviewer に「SPEC 整合」独立 Critical 項目。

M5 — 入口での吸い出し必須化 案A 中核

SPEC.md 無しのレガシーは、clarify 連結 spec-extract（M2）を必須前工程にする。読むだけの推測で始めない。（M5 は案A中核の「吸い出し必須化」で、その入口に要望#5 由来の M2 を内包する。要望対応表では #5＝M2 に一本化し二重計上を避けている。）

9. 検証方法（実装時）

  • 原本と連携版の一致: diff <(awk '/PIPELINE-INTEGRATION/{exit} {print}' software-pipeline/.claude/skills/spec-extract/SKILL.md) implementation-skills/.claude/skills/spec-extract/SKILL.md が空。
  • Agent Skills 規則: 各 SKILL.md 本文が 500 行以内。
  • version 規約: software-pipeline 変更時に plugin.json と marketplace.json の version が更新済み。
  • 手動シナリオ（M4）: SPEC.md を持つ小リポジトリで /feature-pipeline を流し、 (1) researcher が SPEC.md をロード、 (2) SPEC に反する要求が「未解決の質問」で停止、 (3) Phase 7 で SPEC.md が増分更新、 (4) validator が SPEC 整合を独立項目で報告、を確認。
  • M1 小変更同期: pipeline を通さない1ファイル修正で挙動を変えたとき、notes が該当 F-NN を更新し、未同期時にフックが非ブロッキング通知を出す。
  • M2 clarify 連結: /spec-extract 実行時、[不明]/[推定] が残ると一問ずつヒアリングが走り、回答が [確定] に反映される。ヘッドレス/バッチ時は clarify をスキップし、[不明] 一覧が SPEC.md §8（未解決の質問）に成果物として残ることも確認（物証主義を崩さない）。
  • M3 初期化: 空ディレクトリで /pipeline-setup（または /task-pipeline-setup）を実行すると、種別ヒアリング→初期 SPEC 作成へ分岐。既存リポでも生成前に clarify 確認が入る。
