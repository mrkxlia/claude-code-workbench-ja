# task-pipeline — タスクパイプラインオーケストレーター

このスキルは成果物の作成を5つの専門エージェントの流れ作業に変える。
あなた（メインセッション）は**オーケストレーター**であり、自分では成果物を作らない。
各フェーズで Task ツールを使って対応するサブエージェントを起動し、
中間成果物の保存・受け渡し・人間チェックポイントでの停止だけを行う。

## 全体の流れ

```
依頼
 → Phase 1: source-researcher（調査）
 → Phase 2: requirements-writer（成果物要件）   → 🛑 チェックポイント1: 要件承認（Plan モードレビュー）
 → Phase 3: brief-writer（作業ブリーフ）        → 🛑 チェックポイント2: ブリーフ承認（Plan モードレビュー）
 → Phase 4: deliverable-builder（成果物作成）
 → Phase 5: deliverable-reviewer（レビュー）    → 失敗時はビルダーへ差し戻し（上限3回）
                                                → 🛑 チェックポイント3: 最終レビュー
 → コミットの提案（git 管理下の場合）
```

> 凡例: **（Plan モードレビュー）** が付く CP1・CP2 は、Claude Code の Plan モードの計画レビューで承認を取る
> （詳細は後述の「Plan モードレビューについて」）。**CP3 には付かない** — 完成物の検証のため従来どおりテキスト承認。

連鎖が長く、コンテキスト圧縮（コンパクション）やセッション中断で進行状態が失われやすい。
そのため進行状況はコンテキスト内のメモではなく **`docs/task-pipeline/<slug>/status.md` に永続化する**。
Phase 0 で以下のテンプレートを status.md として保存し、フェーズ完了・チェックポイント承認・
差し戻しのたびに Edit で更新すること（フェーズ飛ばしと差し戻し回数の喪失を防ぐ）:

```
# パイプラインの進行状況 — <slug>

- [ ] Phase 0: 準備（slug 決定・docs/task-pipeline/<slug>/ 作成）
- [ ] Phase 1: Research → research.md 保存
- [ ] Phase 2: Requirements → clarify で要件を詰める → requirements.md 保存 → 🛑 要件承認（承認: ／方式: ）
- [ ] Phase 3: Brief → clarify で構成を詰める → brief.md 保存 → 🛑 ブリーフ承認（承認: ／方式: ）
- [ ] Phase 4: Build（セルフチェックに ❌ があれば差し戻し）
- [ ] Phase 5: Review — 差し戻し 0/3 → 🛑 最終レビュー（承認: ）
```

brief に「並列実行プラン」で独立グループが宣言されている場合は、Phase 4 を
「① 共有/先行逐次作成 → ② 独立グループ並列 → ③ 依存グループ逐次」に展開する。

チェックポイントの承認を得たら、その行の「（承認: ）」に日付（YYYY-MM-DD）を記録する
（CP1/CP2 は「方式」に承認方法 — Plan モードレビューなら `Plan`、フォールバックなら `テキスト` — も残す）。
各チェックポイントで停止する際は、status.md の現在の状態をユーザーへの提示に含める。

## Plan モードレビューについて（CP1・CP2）

前向きのチェックポイント **CP1（要件承認）・CP2（ブリーフ承認）** は、Claude Code の
**Plan モードの計画レビュー**で承認を取る。CP3（最終レビュー）は完成物の検証なので対象外（従来どおりテキスト承認）。

パイプラインは research.md / requirements.md / brief.md 等を**書き出す**ため、全体は**通常モード**で動かす。
Plan モードへ入るのは各 CP の提示のときだけにする。Plan モードで起動された場合は CP まで進めない
（Plan モード中は計画ファイルしか書けない）ので、通常モードに戻してから実行する
（戻せなければユーザーに通常モードでの再起動を促す）。

各 CP では次の二段構えで承認を取る:

**(A) Plan モードレビュー（既定 / `EnterPlanMode`・`ExitPlanMode` が使える Claude Code）**
1. 中間成果物を所定パスに保存する（通常モード）。
2. **通常モードであることを確認し、`EnterPlanMode` で Plan モードに入る**（既に Plan モード中なら呼ばない＝冪等）。
3. 計画として次を提示する: ① 承認対象の成果物の**全文**、② **遷移宣言**＝「承認後に実行するのは**次の1フェーズだけ**で、
   完了後に**次の CP で必ず再停止する**」（全工程を今から実行するとは書かない）。
4. `ExitPlanMode` で承認を得る。
5. 承認後は通常モードに戻り、status.md に承認日付・方式を記録して、宣言した**次の1フェーズだけ**を実行する。
   - **「次 CP で必ず止まる」担保**は遷移宣言テキストではなく、**次フェーズ完了時に次 CP として再び
     `EnterPlanMode`→`ExitPlanMode` を呼ぶこと自体**である（承認が出るまで先へ進めない）。
   - 修正指示時は対応ビルダーを再起動し、必要なら再び `EnterPlanMode` に入り直して再提示する。

**(B) フォールバック（`EnterPlanMode`/`ExitPlanMode` が無い環境・非 Claude Code）**
- 成果物の全文を提示し、「🛑 STOP. 明示承認があるまで次フェーズに進まない」で承認を待つ。

> どちらを使うかは、`EnterPlanMode` が使えるかで決める（使えなければ (B)）。

## Phase 0: 準備

1. 依頼内容から短い英語ケバブケースの task-slug を決める（例: `auth-architecture-diagram`）
2. 中間成果物ディレクトリ `docs/task-pipeline/<slug>/` を作成する
3. 上記テンプレートから `status.md` を作成して保存する（依頼が「再開」なら下の「中断からの再開」へ）
4. このディレクトリに以下を保存していく:
   - `status.md` — 進行状況（フェーズ・承認・差し戻しカウンタ）
   - `research.md` — 調査レポート
   - `requirements.md` — 成果物要件
   - `brief.md` — 作業ブリーフ

成果物本体の保存先はここではなく、CLAUDE.md で定めた**出力ディレクトリ**である。

**重要:** researcher / requirements-writer / brief-writer / reviewer は読み取り専用で
ファイルに書き込めない。彼らの出力（テキスト）を `docs/task-pipeline/<slug>/` に保存するのは
あなた（メインセッション）の仕事である。

## Phase 1: Research（調査）

1. Task ツールで `source-researcher` を起動する。入力: ユーザーの依頼内容
2. 返ってきた調査レポートを `docs/task-pipeline/<slug>/research.md` に保存する
3. レポートの「未解決の質問」に依頼内容の理解を左右するものがあれば、ここでユーザーに確認する

## Phase 2: Requirements（成果物要件）→ 🛑 チェックポイント1

1. **clarify で要件を詰める（writer 起動の前に1回）。** `clarify` スキルのプロトコルで、
   research.md を踏まえてもなお人間にしか答えられない要件の曖昧さ・抜けを、**一問ずつ**
   （各問に推奨回答つき・AskUserQuestion を使うなら `questions` は常に1問）ユーザーに確認する。
   調べればわかることは聞かない。掘り尽くしたら「決まったこと」をまとめる。
2. Task ツールで `requirements-writer` を起動する。入力: 依頼内容 + research.md + clarify の「決まったこと」
3. 返ってきた要件を `docs/task-pipeline/<slug>/requirements.md` に保存する
4. ユーザーがファイルを開かずにレビューできるよう、チャットに次の2つを必ず表示する
   （「requirements.md に保存した。確認して」のようにファイルを指すだけで終わらせない）:
   - **要点サマリー** — 依頼の一文要約 + 受け入れ基準の箇条書き
   - **全文** — 保存した requirements.md の内容を省略・要約せずそのまま提示する

**🛑 チェックポイント1 — 「Plan モードレビューについて」の手順で承認を取る。承認🛑はここ1回だけ（clarify を承認のたびに再起動しない）。**
- **(A) Plan モードレビュー**: `EnterPlanMode` →（step 4 の要点サマリー＋requirements.md 全文 ＋ 遷移宣言
  「承認後は brief-writer を起動して作業ブリーフを作り、CP2 で必ず停止する」）を計画として提示 → `ExitPlanMode` で承認を得る。
  承認まで**絶対に Phase 3 に進まない**。
- **(B) フォールバック**: step 4 の要点サマリー＋全文を提示し、明示的な承認（「承認」「OK」「進めて」等）があるまで Phase 3 に進まない。
- 承認後は status.md の「要件承認」に日付と方式（Plan／テキスト）を記録する。
修正指示があれば requirements-writer を修正内容つきで再起動し、再びここで停止する（再提示も step 4 と同じ要点サマリー＋全文の形式で。(A) の場合は再び `EnterPlanMode` に入り直して提示する）。

## Phase 3: Brief（作業ブリーフ）→ 🛑 チェックポイント2

1. **clarify で構成を詰める（writer 起動の前に1回）。** `clarify` スキルのプロトコルで、
   構成上の分岐・優先順位・残る方針判断を**一問ずつ**（各問に推奨回答つき・AskUserQuestion を
   使うなら `questions` は常に1問）ユーザーに確認する。調べればわかることは聞かない。
   掘り尽くしたら「決まったこと」をまとめる。
2. Task ツールで `brief-writer` を起動する。
   入力: 承認済み requirements.md + research.md + CLAUDE.md のルール
   （出力ディレクトリ・利用可能なスキル・表記規約）+ clarify の「決まったこと」
3. 返ってきたブリーフを `docs/task-pipeline/<slug>/brief.md` に保存する
4. ユーザーがファイルを開かずにレビューできるよう、チャットに次の2つを必ず表示する
   （「brief.md に保存した。確認して」のようにファイルを指すだけで終わらせない）:
   - **要点サマリー** — 主要な作業判断・対象成果物・残る論点・**並列実行プランの要旨**
   - **全文** — 保存した brief.md の内容を省略・要約せずそのまま提示する

**🛑 チェックポイント2 — 「Plan モードレビューについて」の手順で承認を取る。この時点では成果物が1ファイルも作成されていないこと。承認🛑はここ1回だけ（clarify を承認のたびに再起動しない）。**
- **(A) Plan モードレビュー**: `EnterPlanMode` →（step 4 の要点サマリー＋brief.md 全文 ＋ 遷移宣言
  「承認後は Phase 4（deliverable-builder）で成果物を作り、Phase 5 のレビューを経て CP3 で必ず停止する」）を計画として提示
  → `ExitPlanMode` で承認を得る。承認まで**絶対に Phase 4 に進まない**。
- **(B) フォールバック**: step 4 の要点サマリー＋全文を提示し、明示的な承認があるまで Phase 4 に進まない。
- 承認後は status.md の「ブリーフ承認」に日付と方式（Plan／テキスト）を記録する。
修正指示があれば brief-writer を再起動し、再びここで停止する（再提示も step 4 と同じ要点サマリー＋全文の形式で。(A) の場合は再び `EnterPlanMode` に入り直して提示する）。

## Phase 4: Build（成果物作成）

1. Task ツールで `deliverable-builder` を起動する。
   入力: 承認済み brief.md + requirements.md + research.md
2. サマリーの**受け入れ基準ごとのセルフチェック**に ❌ がある場合は、完了扱いにせず
   deliverable-builder に差し戻す。❌ の理由が要件側の問題（実現不可能な基準など）であれば、
   ユーザーに報告して指示を仰ぐ
3. サマリーに「ブリーフからの逸脱」が報告されたら、その内容をユーザーへの提示に含める（隠さない）

### Phase 4 の並列実行（brief に「並列実行プラン」で独立グループがある場合）

brief が独立グループを宣言しているときは、Phase 4 を次の順で実行する。宣言が無ければ
上の既定どおり単一の deliverable-builder で逐次作成する。

1. **① 共有/先行逐次作成** — 全体目次・用語集・全体索引・共通テンプレートなど共有部分を、
   まず1つの deliverable-builder で逐次作成して固定する。
2. **② 独立グループを並列起動** — 独立グループの deliverable-builder を**1メッセージ内の複数 Task で
   同時に**起動する。各ビルダーへの入力は「承認済み brief + **当該グループの所有パスだけ**
   （brief 全体を丸投げしない）」。各グループは出力ディレクトリ内の交わらないサブパスを所有する。
3. **③ 依存グループを逐次** — 依存エッジに従い、依存先（例: 図を引用するレポート）を逐次作成する。
4. 越境（あるグループのビルダーが他グループの所有パスへ書く）を見つけたら、その変更を破棄して
   正しい担当に差し戻す。
   - **注意:** 既存の `guard-deliverable-writes.sh` フックが守るのは「出力ディレクトリ*外*」への
     書き込みだけで、出力ディレクトリ*内*のグループ境界は守らない（許可リスト内一致で素通り）。
     グループ境界は brief の所有宣言＋オーケストレーターの越境チェックで守る。

## Phase 5: Review（レビュー）→ 🛑 チェックポイント3

1. Task ツールで `deliverable-reviewer` を起動する。
   入力: requirements.md + brief.md + ビルダーのサマリー
2. レポートに **Critical** または **Important** の指摘があれば:
   - deliverable-builder を指摘内容つきで再起動する。自分で直さない。reviewer にも直させない
   - 差し戻すたびに status.md の Review カウンタを更新する（例: `差し戻し 1/3`）。
     コンテキストの記憶ではなく status.md のカウンタを正とする
   - 修正後、deliverable-reviewer を再実行する
3. **差し戻しループの上限は3回。** 超えたら停止し、状況をユーザーに報告して指示を仰ぐ
4. **学習の回収:** ビルダーのサマリーに「CLAUDE.md への提案」があれば、
   `docs/task-pipeline/LEARNINGS.md` に追記する
   （形式: `- [ ] YYYY-MM-DD <slug> (deliverable-builder): 提案内容`）
5. クリーンになったら、最終レポート・成果物ファイル一覧・LEARNINGS.md の未昇格エントリ
   （チェックの付いていない行）をユーザーに提示する

**STOP. ユーザーがレビューして承認するまで、完成・コミットに進んではならない。**
承認時、提示した LEARNINGS エントリを CLAUDE.md のルール・規約に昇格させるか確認する。
昇格させたエントリには LEARNINGS.md 上でチェック（`- [x]`）を付ける（採用しないものは未チェックのまま残す）。
**生きた成果物仕様の更新（SPEC.md がある場合）:** 承認された成果物が既存の内容・構成・規約を
追加/変更/廃止したなら、`spec-extract` スキルの「変更管理」に従い SPEC.md（成果物仕様）を**増分更新**する
（新規=新 `D-NN` 追加、変更=同 ID で書き換え＋§9 改訂履歴に1行、廃止=`[廃止]` 印）。brief の
「既存仕様への影響」で宣言された差分が反映対象。SPEC.md が無ければスキップしてよい（必要なら `/spec-extract` を案内）。
承認後、git 管理下ならコミット（およびユーザーが望めばPR作成）を提案する。
git 管理されていないプロジェクトでは、成果物ファイル一覧の提示をもって完了とする。

## 中断からの再開

セッション中断やコンパクションで進行状態を見失ったら、会話の記憶ではなく
`docs/task-pipeline/<slug>/status.md` を正として再開する:

1. `/task-pipeline 再開 <slug>`（または slug を特定できる再開の依頼）を受けたら、
   `docs/task-pipeline/<slug>/` の status.md と保存済みの中間成果物
   （research.md / requirements.md / brief.md）を読む
2. status.md の最初の未チェックフェーズから処理を続ける
3. 承認が記録されているチェックポイントは再承認を求めない（承認済みの成果物をそのまま入力に使う）
4. status.md と現実が食い違う場合（例: Phase 3 がチェック済みなのに brief.md が無い）は、
   食い違いをユーザーに報告し、整合する最後のフェーズからやり直す

## オーケストレーターのルール

- チェックポイントのスキップは、ユーザーが事前に明示的に指示した場合のみ許される
- 各エージェントには必要な中間成果物だけを渡す。会話履歴全体を流し込まない（コンテキストを汚さない）
- エージェントの担当範囲の越境（例: reviewer が成果物を直す、builder が出力ディレクトリの
  外に書く）を見つけたら、その変更を破棄して正しい担当者に差し戻す
- 途中で要件の前提が間違っていたと判明したら、パッチで誤魔化さず、
  正しい前提を織り込んで該当フェーズからやり直す
