# long-run — 長時間自律作業の完走プロトコル

上位モデル（Fable 5）は数時間規模の作業を、途中で許可を求めず・進捗を捏造せず・
早期に切り上げずに完走する。このスキルはその振る舞いを Opus / Sonnet で再現するための
明示的なプロトコルである。文面の多くは公式 Prompting Claude Fable 5 ガイドの
自律実行・進捗監査プロンプトの翻案。

## 使いどころ／住み分け

- 対象: リファクタ一式・一括移行・大量ドキュメント作成など、着手後はユーザーの判断が
  ほぼ不要な長い作業を任されたとき
- ステップごとに承認を取りながら進めたい → `backlog-loop`（両方が起動している場合は
  backlog-loop の承認ゲートが優先。本スキルの自律ブロックはゲートを上書きしない）
- 変更せず計画だけ作る → `create-plan`

## 中核ルール

0. **開始条件: 完了条件が確定していること。** 未確定なら先に `task-brief` で確定させる。
   承認されたブリーフ（完了条件・スコープ・制約）がこのプロトコルの憲法になる。

1. **自律ブロック。** 着手後は次の規律で動く:
   あなたは自律運転中であり、ユーザーはリアルタイムで見ていない。「〜しましょうか?」
   「進めてよいですか?」という確認質問でターンを終えると作業全体が止まる。
   承認済みブリーフの範囲内で可逆な操作は、聞かずに実行する。
   **ターンを終える前に、自分の最後の段落を見直す。それが計画・分析・質問・
   次のステップの列挙・「これから〜します」という約束なら、ターンを終えずに
   いまツールでその作業を実行する。**

2. **停止条件は次の4つだけ（閉じた列挙）。** これ以外の理由で停止しない:
   - 成果物が変わる複数解釈が出た（ルール2）
   - 同じエラーへの修正が2回失敗した（ルール5）
   - スコープの変化・拡張が必要になった（ルール8）
   - 破壊的・不可逆な操作（削除・上書き・push・外部送信）の前提が証拠で確認できない

3. **区切りごとの進捗報告は証拠監査つき。** 主要な成果物の完成・フェーズの切り替わりごとに、
   「完了したこと／次にやること／気になっていること」の3点（ルール7）を報告する。
   報告する各主張は、このセッションのツール実行結果と突き合わせてから書く（追補ルール10）。
   証拠を指させない項目は「未検証」と明記する。
   **報告の間隔を機械的に固定しない**（「nツール呼び出しごとに報告」のような強制は、
   最新モデルでは不要というのが公式ガイダンス。区切り基準で報告する）。

4. **検証チェックポイント。** 主要な成果物が完成するごとに、`fresh-verifier`（/verify-fresh）に
   成果物と完了条件を渡して検証させる。自己批評よりも、経緯を知らない fresh context の検証の
   ほうが問題を見つける — 自分のレビューで代替しない。

5. **序盤制約の保持。** 承認されたブリーフ（完了条件・スコープ・制約）を作業メモ
   （TodoWrite または作業ディレクトリのメモファイル）に固定し、**各区切りで再読してから**
   次の作業に進む。長い作業では序盤の制約が薄れるため、構造で保持する。

6. **最終サマリは経過を見ていない読者向けに書く。** 結論（何が完了し、何が残ったか）を
   先頭に、完全な文で。作業中に自分が作った略語・ラベル・矢印つなぎを持ち込まない。

## ワークフロー

1. 完了条件を確認（なければ task-brief）。ブリーフを作業メモに固定して開始を宣言する。
2. 自律ブロックの規律で作業を進める。区切りごとに: 進捗報告（証拠監査つき）→ ブリーフ再読 → 続行。
3. 主要成果物の完成ごとに fresh-verifier で検証。Critical は修正して再検証（同一問題は2回まで）。
4. 完了条件をすべて満たしたら、最終サマリを出して終了する。

## 出力テンプレート

**開始宣言**

```
## long-run 開始
- 完了条件: <ブリーフから再掲>
- スコープ: <触る範囲 / 触らない範囲>
- 停止するのは: 複数解釈 / 同エラー2回 / スコープ変化 / 不可逆操作の前提未確認 のみ
```

**区切り報告**

```
## 区切り報告 (n)
- 完了したこと: <各項目に証拠（コマンド・出力・ファイル）を添える。未検証は「未検証」と明記>
- 次にやること: ...
- 気になっていること: ...
```

**最終報告**

```
## long-run 完了
<結論を1〜2文で: 何が完了し、何が残ったか>
- 完了条件との対応: <条件ごとに 満たした証拠 / 未達なら理由>
- 検証: <fresh-verifier の判定と残指摘（全件）>
- 残作業・提案: ...
```

## やらないこと

- 停止条件（中核ルール2）に該当しない場面での確認質問・許可待ち
- 「問題なく進んでいます」だけの報告、証拠のない完了報告
- 同じエラーへの3回目の修正の変種
- コンテキストの長さを理由にした要約・切り上げ・「新しいセッションで続けましょう」の提案

## 連携

- 着手前の完了条件確定 → `task-brief`
- 区切りの検証 → `verify-fresh`
- 分割できる部分の並列化 → `fan-out`（その間も本スキルの報告規律は維持）
- 教訓の記録 → knowledge-share プラグインの `kb`（このスキルでは記録の仕組みを持たない）
