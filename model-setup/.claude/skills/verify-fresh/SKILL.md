# verify-fresh — fresh context 検証

作業した本人による自己レビューは、経緯と自己評価に引きずられて楽観に偏る。
公式ガイダンスも「自己批評より、fresh context の検証サブエージェントのほうが有効」としている。
このスキルは、成果物と完了条件**だけ**を独立サブエージェント `fresh-verifier` に渡し、
反証的に検証させる定型手順である。ルール6（初見レビュー）のサブエージェント強化版にあたる。

## 使いどころ／住み分け

- 対象: 完了報告の前・マージの前・引き渡しの前に、成果物が完了条件を本当に満たすか確かめたいとき。
  `fan-out` の成果マージ前と `long-run` の区切りでは標準工程として呼ばれる
- 実装**前**の計画・設計の相談 → `peer`（ai-peer）
- Codex による第二の目（コードレビュー） → `codex-review`（codex-bridge）
- feature-pipeline 実行中のストーリー/ブリーフ突き合わせ → `implementation-validator`（同パイプライン Phase 7）

## 中核ルール

1. **verifier には成果物の場所と完了条件だけを渡す。** 作業の経緯・自分の自己評価・
   「ここは大丈夫なはず」という情報は渡さない。fresh context の価値はアンカリングの不在にある。
2. **反証フレーミングで依頼する。**「問題ないか見て」ではなく「これを完了と認めない理由を探せ」。
3. **網羅指示を必ず入れる。** verifier への依頼に次の指示を含める（Sonnet は「重要なものだけ
   報告して」と言われると本物の問題まで自己選別して落とすため、公式ガイダンスの補正を使う）:
   「見つけた指摘は、確信が持てないもの・低重要度と思うものも含めて全件報告すること。
   重要度や確信度でこの段階では絞り込まない — フィルタリングは後工程が行う。
   目標は網羅であり、後で捨てられる指摘を出すほうが、本物の問題を黙って落とすより良い。
   各指摘には重大度（Critical/Important/Minor）と確信度（高/中/低）を付けること。」
4. **verifier は検証コマンドを実行してよいが、修正はしない。** 修正はこちら（呼び出し側）の仕事。
5. **Critical は修正 → 再検証。** 同じ指摘への修正が2回失敗したら3回目を試さず、ユーザーに
   報告する（ルール5）。Important / Minor は修正せず全件をユーザーへの報告に載せ、判断を委ねる。

## ワークフロー

1. 完了条件を特定する（引数のパス → ブリーフ/SPEC → 直近の会話で確定した完了条件、の順で探す。
   どこにも無ければ、検証の前に完了条件を1行で確定させる）。
2. `fresh-verifier` サブエージェントを起動し、下記テンプレートで依頼する。
3. 判定を受け取り、Critical があれば修正して再検証（中核ルール5の上限内）。
4. 結果をユーザーに報告する（指摘は Minor まで全件・重大度つき。ルール9）。

## 出力テンプレート

**verifier への依頼**

```
成果物: <パス・場所>
完了条件: <機械的に判定できる基準（複数可）>
参照してよい仕様: <あれば SPEC.md・ブリーフのパス>

この成果物を「完了と認めない理由」を探してください。実行できる検証（テスト・ビルド・
コマンド）は実際に実行し、出力を証拠として記録してください。
見つけた指摘は、確信が持てないもの・低重要度と思うものも含めて全件報告すること。
重要度や確信度でこの段階では絞り込まない — フィルタリングは後工程が行う。
各指摘には重大度（Critical/Important/Minor）・確信度（高/中/低）・証拠（path:line または
コマンド出力）を付けること。修正はしないこと。
```

**ユーザーへの報告**

```
## verify-fresh 結果: <完了と認められた / Critical n件>
- 実行された検証: <コマンド → 結果>
- 指摘（全件・重大度つき）: ...
- 対応: <Critical の修正と再検証結果 / 未対応の指摘と理由>
```

## やらないこと

- verifier への経緯・自己評価の受け渡し（アンカリングのもと）
- 指摘の勝手な間引き（Minor だから報告しない、はルール9違反）
- verifier の指摘の無検証な鵜呑み — Critical の修正前に指摘自体の証拠を確認する
- 完了条件が無いままの「なんとなく検証」

## 連携

- 並列作業の成果マージ前 → `fan-out` の手順4として呼ばれる
- 長時間作業の区切り → `long-run` の検証チェックポイントとして呼ばれる
- 完了条件そのものが無い → 先に `task-brief` で確定させる
