# task-brief — 最初のターンで仕様を完成させる

Claude Sonnet 5 は指示を字義どおりに実行し、依頼を勝手に一般化しない代わりに、範囲の明示が
無い項目は埋めてくれない。加えて、依頼を複数ターンに分けて小出しにすると効率が落ちる
（公式ガイダンス: 最初のターンでタスク・意図・制約を完全に指定するほど、自律性と効率が上がる）。
このスキルは、着手前の1往復で「完全なタスク仕様」を確定させることでこの2点を同時に補う。

## 中核ルール

1. **不足項目は一括で質問する。** 1問ずつの往復は避け、埋まっていない項目をまとめて、
   それぞれに推奨値を添えて確認する。
2. **どの解釈でも結果が同じ項目は聞かない。** 曖昧さがあっても成果物に影響しないなら、
   質問を増やさず妥当な解釈で進める。
3. **ブリーフが確定したら、それが以降の作業の正になる。** 作業中に指示が変わったら、
   ブリーフ（特に完了条件・スコープ）を更新してから再掲する。
4. **5問を超える質問が必要なら、このスキルの範囲を超えている。** clarify に委ねる。

## ワークフロー

1. 依頼を以下の6項目に分解する: ゴール／完了条件／スコープ（対象・非対象）／制約／検証方法／報告形式。
2. 埋まらない項目を洗い出す。各項目に推奨値（こう解釈して進めてよいか）を添えて、
   まとめて1回のメッセージで確認する。
3. 回答（または「推奨のままで」の一言）を受けて、ブリーフを完成させて提示する。
4. ユーザーの承認を得る。
5. 「このブリーフを完了条件とする」と宣言してから着手する。

## 出力テンプレート

```markdown
## ゴール
（1〜2行。何のために何をするか）

## 完了条件（機械的に判定できる形で）
- [ ] ...

## スコープ
- 対象: ...
- 非対象: ...

## 制約
- ...

## 検証方法
- ...

## 報告形式
- 完了時は検証の証拠（実行コマンド・戻り値・テスト結果等）を添える
- 確信度が中・低の箇所には確信度ラベルを付ける
```

## やらないこと

- ブリーフ確定前に実装へ進まない。
- 5問を超える質問攻めをしない（→ clarify に委ねる）。
- 読み取り専用の実行計画ファイル（別セッション/別エージェントへの引き継ぎ用）は作らない
  （→ create-plan）。

## 連携

- 大きな機能開発・複雑な要件は、ブリーフ確定後に `create-plan` で計画ファイル化してから着手すると
  Opus で計画・Sonnet で実行という分担がしやすい（`model-setup/MODEL-GUIDE.md` 参照）。
- backlog.md 駆動の定型ループでは、`backlog-loop` の Step0 として本スキルを使うと、
  各 Step の完了条件が最初から明確になる。
- 要件が固まらない・利害関係者ごとに答えが割れるような複雑な意思決定には `clarify` を使う。
