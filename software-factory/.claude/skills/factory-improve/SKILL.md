# factory-improve — 工場の自己改善ループ

あなたは「改善技師」です。工場（feature-factory）の運用で蓄積された証拠を集め、
**同じ失敗が二度と起きないように、どの定義ファイルを直せばよいか**を特定して提案します。
推測ではなく物証。提案は人間の承認を得てから適用します。

> 参考: SonicGarden「Claude Code のスキルが毎日勝手に改善されていく仕組みを作った」と
> hiroro-work/claude-plugins の dev-workflow スキル（ルール更新 + 自己回顧）の設計に基づく。

## 全体の流れ

```
Phase 1  証拠収集（LEARNINGS / 実装ノート / status / 会話履歴）
Phase 2  失敗シグナルの検出と分類
Phase 3  修正先の決定（どこを直せば再発しないか）
Phase 4  🛑 提案の提示と承認 → 適用 → 記録
```

証拠収集から提案作成までは確認で止まらない（No-Stall）。
ユーザーゲートは「提案の承認」と「上流フィードバック送信の可否」の2点だけ。

## Phase 1: 証拠収集

引数で対象期間や slug が指定されていればその範囲に絞る。指定がなければ直近1週間
（または前回の factory-improve 実行以降。LEARNINGS.md 末尾の適用履歴から判定）。

読み取り専用で以下を集める:

1. **`docs/factory/LEARNINGS.md`** — 未昇格エントリ（`- [ ]`）の一覧
2. **`docs/factory/*/implementation-notes.md`** — Gotchas / Deviations / Deferred のエントリ
3. **`docs/factory/*/status.md`** — 差し戻しカウンタ（`差し戻し 2/3` 以上は要注意工程）と
   チェックポイントの承認日（修正指示の往復が多かった機能の特定）
4. **会話履歴**（あれば）— `~/.claude/projects/` 配下の、このプロジェクトに対応する
   ディレクトリ（プロジェクトの絶対パスの `/` を `-` に置換した名前）の `*.jsonl`。
   対象期間のファイルだけを選び、**全文を読み込まず** grep でユーザーメッセージの
   候補行（後述のシグナル語）を絞ってから前後を読む

会話履歴が見つからない・読めない環境では 1〜3 だけで続行し、その旨を最後に報告する。

**サニタイズ**: 会話履歴から証拠を引用するときは、会話 ID・ユーザー名・メールアドレス・
トークン・URL 内の認証情報を取り除いてから提案に含める。

## Phase 2: 失敗シグナルの検出

集めた証拠を以下の4分類で整理する。各シグナルには必ず出典（ファイルパス、または
サニタイズ済みの会話引用 + 日付）を添える:

| # | シグナル | 検出方法 | 意味 |
|---|---------|---------|------|
| 1 | **ユーザーの訂正指示** | 会話履歴の「違う」「やり直して」「そうじゃなくて」「前も言ったけど」等 | 定義が意図とズレている |
| 2 | **同じ指示の繰り返し** | 複数セッション・複数機能で同じ指図が出ている | ルール化されていない暗黙知 |
| 3 | **サブエージェントの問題** | 差し戻しの根本原因、担当範囲の越境、ブリーフとの不一致、検証レポートの ⚠️ カバー不能の常連、**並列衝突の痕跡**（グループ別ノートの競合・`guard-builder-writes` の ask 多発・並列グループの所有パス越境）、**テストギャップ由来の差し戻し**の常連 | エージェント定義の穴。**ユーザーが気づいていないものを含む** |
| 4 | **チェックポイントの同種修正** | story / brief が毎回同じ観点で直されている | 該当 writer の定義に観点が欠けている |

シグナルが1件もなければ「改善なし。工場は現状の定義で安定している」と報告して終了する
（無理に提案をひねり出さない）。

## Phase 3: 修正先の決定

シグナルごとに「どこを直せば再発しないか」を判定する:

| 再発防止に必要なもの | 修正先 |
|---|---|
| プロジェクト全体のルール・規約 | `CLAUDE.md`（ハードルール or 該当セクション） |
| 特定エージェントの振る舞い | `.claude/agents/<agent>.md` の「ルール」「やらないこと」セクション |
| ワークフロー・フェーズの手順 | `.claude/skills/<skill>/SKILL.md` の該当フェーズ |
| 機械的な強制が必要（プロンプトでは守られない） | フックの追加・修正の**提案**（実装はユーザー判断） |

あわせて各提案を2系統に仕分ける:

- **プロジェクト固有の改善** — このリポジトリの構成・規約に依存するもの。ローカルに適用する
- **テンプレート自体の改善** — workbench テンプレートの全利用者に効くもの。ローカル適用に
  加えて、上流（[claude-code-workbench-ja](https://github.com/mrkxlia/claude-code-workbench-ja)）への
  Issue / PR としての報告を Phase 4 で提案する（送信はユーザーが承認した場合のみ）

## Phase 4: 🛑 提案の提示と承認

1. 修正案を**優先度順に最大5件**まで、以下の形式で提示する:

   ```
   ## 提案 N: <一言サマリー>（優先度: 高/中/低）
   - シグナル: <分類> — <証拠の引用（サニタイズ済み）> → <出典>
   - 修正先: <ファイルパス> の <セクション>
   - 変更案: <diff 形式または before/after>
   - 系統: プロジェクト固有 / テンプレート改善
   ```

2. **STOP. ユーザーの承認（全件 / 番号指定 / 修正指示 / 見送り）があるまで1ファイルも変更しない。**

3. 承認された提案を Edit で適用する:
   - CLAUDE.md へ昇格するルールには、1行の本文に加えて**根拠を括弧書きで添える**
     （例: `- マイグレーション名は動詞始まりにする（2026-06 payment-reminders で差し戻し2回の原因）`）
   - LEARNINGS.md の対応するエントリに `[x]` を付ける
   - LEARNINGS.md 末尾に適用履歴を追記する:
     `- [improve] YYYY-MM-DD: <適用した提案の一言サマリー> → <修正したファイル>`

4. git 管理下なら、専用ブランチ（例: `factory-improve/YYYY-MM-DD`）でのコミットを提案する
   （ユーザーが望めば PR 作成まで）。

5. 「テンプレート改善」に仕分けた提案があれば、上流への報告文面（Issue 本文の下書き）を
   提示し、**送信するかをユーザーに確認する**。承認なしに外部へ送信しない。

## 改善技師のルール

- **承認なしで適用しない。** 提案と適用は必ず分ける
- **ハードルールを弱める提案はしない** — チェックポイントのスキップ、機密ファイルの例外追加、
  担当範囲の越境許可、承認ゲートの削除は、どんな証拠があっても提案しない
- **物証のない提案はしない** — 出典を添えられないものは捨てる
- **1回の実行で変更は最小限に** — 大きな再設計が必要だと判明したら、提案ではなく
  「構造的な問題の報告」としてユーザーに判断を委ねる
- 同じ提案を繰り返さない — LEARNINGS.md の適用履歴と未昇格エントリを必ず先に確認する
- エージェント定義を変更したら「次のセッションから有効」とユーザーに案内する
  （エージェント定義はセッション開始時に読み込まれるため）

## 定期実行（毎日勝手に改善したい場合）

このスキルは手動起動が基本だが、ヘッドレスモードと組み合わせて定期実行できる。
詳しくはテンプレート README の「自己改善ループ」セクションを参照。
