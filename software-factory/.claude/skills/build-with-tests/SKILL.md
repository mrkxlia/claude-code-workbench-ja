# build-with-tests — テストと並行して実装するスキル

このスキルは「うちのチームのコードの作り方」を定義する。
実装を依頼されたら、以下の3フェーズを順番に、省略せずに実行すること。

## フェーズ1: 書く前に（既存パターンに合わせる）

1. これから書くものに似た既存コードを Grep / Glob で探す
2. **最低3件**の実例を読み、次を確認する:
   - 命名規則（ファイル名・関数名・変数名）
   - エラー処理のパターン（例外か Result か、どこでログを出すか）
   - テストの書き方（配置場所、describe/it の構成、モックの流儀）
3. 既存のヘルパー・ユーティリティで再利用できるものをリストアップする
4. 類似コードが3件見つからない新規領域の場合は、その旨をユーザーに伝えてから書き始める

## フェーズ2: 書いている間（実装とテストを並行）

1. 実装とテストを**同一ターンで**作成する。テストを後回しにしない
2. テストは正常系だけでなく、失敗系・境界値を必ず含める
3. フェーズ1で見つけた既存パターンに従う。我流のパターンを持ち込まない
4. 新しい依存パッケージは、ユーザーの明示的な指示がない限り追加しない

## フェーズ3: 書いた後（検証）

1. 書いたテストを実行し、すべて通ることを確認する
2. 影響範囲の既存テストも実行し、壊していないことを確認する
3. **最後に必ず型チェックを走らせる**:

   ```bash
   npm run typecheck   # ← プロジェクトのコマンドに差し替える（例: tsc --noEmit, mypy .）
   ```

4. Lint があれば実行する

## Done の定義

以下がすべて満たされて、はじめて「完了」と報告してよい:

- [ ] 新しいテストがすべて緑
- [ ] 既存テストを壊していない
- [ ] 型チェックが通っている
- [ ] 既存パターンに従っている（フェーズ1の確認結果と矛盾しない）

1つでも満たせない場合は、「完了」ではなく「何が残っているか」を報告する。
