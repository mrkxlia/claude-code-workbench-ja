## ハードルール（常に適用）

1. **機密ファイルは絶対にコミットしない** — `.env`・`*.key`・`*.pem`・`secrets.json` をステージしない（pre-commit フックでもブロックされる）
2. **担当領域を越境しない** — バックエンド担当はフロントエンドのファイルに触れない。逆も同じ。テスト担当はプロダクトコードを直さない
3. **人間承認チェックポイントを飛ばさない** — ストーリー承認・ブリーフ承認・最終レビューの3点では、明示的な承認があるまで先に進まない
4. **新しい依存パッケージを勝手に追加しない** — 必要なら提案して承認を待つ
5. **テストと型チェックを走らせずに「完了」と言わない**
6. **ビジネスルールを推測で埋めない** — 不明点は「未解決の質問」として人間に確認する

## アーキテクチャのルール

<!-- 差し替え: フォルダ境界は .claude/agents/ のビルダー定義の「担当範囲」と一致させること -->

- **ビジネスロジックはサービス（`src/server/`）に置く。APIルート（`src/app/api/`）は薄く保つ**
- フロントエンドは `src/app/`（`api/` を除く）・`src/components/`・`src/hooks/`
- バックグラウンドジョブは `src/jobs/` に置き、**cron を足さず BullMQ を使う**
- テストは `tests/`（受け入れ・E2E）と実装隣接の `*.test.ts`（ユニット）
- すべてのクエリはテナント（組織）IDでスコープする — テナント分離をすり抜けるクエリを書かない
- 日時は UTC で保存し、表示時にユーザーのタイムゾーンへ変換する

## やってはいけないこと

- 生の決済ペイロード・個人情報・トークンをログに出さない
- 生のエラーオブジェクトをクライアントに返さない（メッセージを整形する）
- マイグレーションファイルを後から手で書き換えない（新しいマイグレーションを足す）
- `any` 型で型エラーを黙らせない

## 7人のAI社員（サブエージェント）

機能開発は `/feature-factory` で以下の連鎖に流す。各エージェントの定義は `.claude/agents/` にある。

| エージェント | 役割 | 書き込み範囲 |
|-------------|------|-------------|
| `codebase-researcher` | 作る前にコードをマッピングする | なし（読み取り専用） |
| `story-writer` | アイデアを受け入れ基準つきストーリーにする | なし（読み取り専用） |
| `spec-writer` | ストーリーを技術ブリーフにする | なし（読み取り専用） |
| `backend-builder` | API・サービス・ジョブ・ユニットテスト | バックエンドのフォルダ + 実装ノート |
| `frontend-builder` | コンポーネント・ページ・フック・UIテスト | フロントエンドのフォルダ + 実装ノート |
| `test-verifier` | ストーリーに対する受け入れテスト | テストファイル + 実装ノート |
| `implementation-validator` | 実装とストーリー/ブリーフのギャップ報告 | なし（読み取り専用） |

## 利用可能なスキル

| スラッシュコマンド | 使用する場面 |
|-------------------|-------------|
| `/feature-factory <機能の説明>` | 機能を end-to-end で開発する（7エージェント連鎖 + 3チェックポイント） |
| `/clarify <詰めたい要件>` | 要件・仕様を一問ずつ徹底質問で詰める（工場内では Phase 2/3 の writer 起動前に自動で回る） |
| `/build-with-tests <タスク>` | 工場を通すほどではない小さな実装・修正をテスト並行で行う |
| `/spec-extract [対象パス]` | 仕様書のないレガシーコードの挙動を SPEC.md に逆引き固定する（工場導入の前工程） |
| `/notes` | 実装ノートを手動で開始・更新する（工場内ではビルダーが自動記録） |
| `/factory-improve [期間や slug]` | 運用実績から失敗シグナルを検出し、エージェント定義・スキル・CLAUDE.md の改善案を提案・適用する |

## 並列実行・テストギャップ・調査の網羅性（補足）

- **並列実行グループ** — brief の「並列実行プラン」が独立グループ（所有パスが交わらず共有ファイルを書かない）を
  宣言したときだけ、Phase 4/5 を「共有先行逐次 → 独立グループ並列 → 依存逐次」で実行する。並列中の共有ファイル
  衝突は `guard-builder-writes.sh`（PreToolUse）が `ask` で確認する。グループ境界の越境は工場長が守る
- **テストギャップ分析** — Phase 6 で test-verifier が、brief の「必要なテスト」（成功/失敗/境界/エラーパス/エッジ）の
  未カバーを埋める。コード修正が要るギャップは担当ビルダーへ差し戻す（上限3回に計上）
- **調査の網羅性** — codebase-researcher は変更に関係する範囲に限り、呼び出し連鎖・依存・既存実装を
  網羅的に読む（表面的サンプリングで済ませない）。コードを書く前の現状把握を徹底する
