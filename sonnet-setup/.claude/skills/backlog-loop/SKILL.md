# backlog-loop — backlog.md 駆動の定型ループ

「backlog.md を読んで計画→実施→完了処理→backlog 更新」という一連の作業を、指示を毎回
組み立て直さずに回すためのスキルです。**各ステップ完了ごとに必ず停止して承認を待ちます**
（自動で次のステップへ進みません）。

## 中核ルール

1. **backlog ファイルが指定・特定できなければ、まず場所を確認する。** 既定は
   プロジェクト直下の `docs/backlog.md`（無ければ `backlog.md`）。
   **self-improve の improvement-backlog（改善提案のキュー）は対象外** —
   そちらは `/improve-apply` に任せる。
2. **計画は Step に分割し、各 Step に完了条件を持たせる。** タスク仕様が曖昧なら、
   Step0 として `task-brief` 相当のブリーフ化を行ってから Step 計画を立てる。
3. **各 Step 完了ごとに必ず停止し、承認を待つ。** 「OK」「進めて」「Step2へ」等の
   承認語を受けてから次の Step へ進む。承認なしに複数 Step を跨がない。
4. **完了処理は環境で分岐する。**
   - git リポジトリ内で `gh` が使える → 全 Step 完了後、`pr-merge` スキルへ委譲する
     （コミット分割〜PR〜マージ〜後片付け）。
   - git が無い（または `git rev-parse --is-inside-work-tree` が失敗する）→
     検証結果つきの完了報告と、変更したファイルの一覧を提示する（持ち出し・バックアップ・
     zip 化の材料として使えるように）。
5. **backlog 更新は本スキルの責務。** マージ／完了処理のあと、該当タスクを完了マーク
   （日付・あれば PR 番号）に更新し、次の候補を提示する。

## ワークフロー

1. backlog ファイルを読み込む。対象タスクを特定する（複数候補があれば確認する）。
2. Step 計画を提示し、承認を得る。
3. Step を1つ実行し、検証つきで報告する（下記テンプレート）。
4. ユーザーの承認を待つ。承認が得られたら次の Step へ。得られなければ止まって相談する。
5. 全 Step 完了後、環境に応じた完了処理（ルール4）を行う。
6. backlog ファイルを更新する（完了マーク・日付・PR番号など）。
7. 次のタスク候補を提示する。

## Step 完了報告テンプレート

```markdown
### 完了したこと
- ...（検証の証拠: 実行コマンド・戻り値・テスト結果等）

### 次にやること
- ...

### 気になっていること
- ...（確信度が中・低の箇所があれば明記）
```

## やらないこと

- 承認なしに複数 Step を跨いで進めない。
- backlog に無い「ついで」の作業をしない。
- マージ処理そのものを重複実装しない（→ `pr-merge` に委譲）。

## 連携

- Step0 のタスク仕様確定には `task-brief` を使う。
- 完了処理（PR〜マージ）は `pr-merge` に委譲する（git/gh が使える環境のみ）。
- 大きすぎるタスクは、着手前に `create-plan` で計画ファイル化することも検討する。
