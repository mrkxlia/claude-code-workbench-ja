# fan-out — 独立サブタスクの並列委譲と検証マージ

上位モデル（Fable 5）は独立サブタスクを自発的にサブエージェントへ並列委譲し、委譲中も
自分の作業を続け、成果を検証してからマージする。このスキルはその振る舞いを Opus / Sonnet で
再現するための明示的な手順である。

## 使いどころ／住み分け

- 対象: 複数ファイルの一括修正、複数ドキュメントの並行作成、独立した調査の同時実行など、
  **書き込み範囲が交わらないサブタスクに分割できる**タスク
- 機能開発（ストーリー→ブリーフ→実装→検証の連鎖が要るもの）→ `feature-pipeline`
- backlog.md 駆動の定型ループ → `backlog-loop`
- 計画だけ作って別セッションで実行 → `create-plan`（本スキルは同一セッション内で完結する）

## 中核ルール

1. **独立性の判定が最初の仕事。** 書き込みファイル集合が交わらず、実行順序の依存がない
   サブタスクだけを並列化する。交わる・依存があるものは逐次に回す（判定に迷ったら逐次）。
2. **worker にはブリーフ6項目を渡し切る。** ゴール／完了条件／スコープ（触ってよい・
   いけないファイル）／制約／検証方法／報告形式。Sonnet は字義どおりに実行するため、
   渡していない判断を worker に期待しない。
3. **委譲中も手を止めない。** worker の完了を待つだけの時間を作らず、自分は逐次分の
   サブタスクやマージ準備を進める。
4. **成果は fresh-verifier で検証してからマージする。** 各 worker の報告を鵜呑みにせず、
   完了条件との突き合わせは fresh context の `fresh-verifier` に任せる（自己申告の検証は弱い）。
5. **失敗した worker の再委譲は2回まで。** 2回失敗したサブタスクは自分で引き取るか、
   ユーザーに報告して判断を仰ぐ。
6. **並列数は既定で最大3。** それを超える分解になる場合は、分解案の承認時に並列数も明示して
   承認を得る。
7. **前処理の対象洗い出しは bulk-scanner に流せる。**「該当ファイルの一覧化」など機械的な
   スキャンは haiku の `bulk-scanner` で安価に済ませる。

## ワークフロー

1. タスクをサブタスクに分解し、**分解案**を提示する（下記テンプレート）。
2. 🛑 **ユーザーの承認を待つ。承認前に worker を起動しない。**
3. 並列グループの worker を一斉に起動する（各 worker にブリーフ6項目）。逐次分・依存分は
   自分で進める。
4. worker の報告を収集し、各成果を `fresh-verifier` にかける（成果物の場所と完了条件だけを渡す）。
5. verifier の Critical 指摘は該当 worker に差し戻す（ルール5の上限内）か自分で対処する。
6. 全サブタスクの検証が終わったらマージし、最終報告を出す。

## 出力テンプレート

**分解案（承認用）**

```
## fan-out 分解案
| # | サブタスク | 完了条件 | 書き込み範囲 | 依存 | 実行 |
|---|-----------|---------|------------|------|------|
| 1 | ... | ... | src/a/** | なし | 並列 |
| 2 | ... | ... | src/b/** | なし | 並列 |
| 3 | ... | ... | 共有 config | #1,#2 | 逐次（最後） |
並列数: n / 想定される衝突: なし（書き込み範囲は互いに素）
この分解で進めてよいですか?
```

**最終報告**

```
## fan-out 結果
| # | サブタスク | 結果 | 検証（fresh-verifier） |
|---|-----------|------|----------------------|
| 1 | ... | 完了 | 問題なし（実行: npm test → exit 0） |
| 2 | ... | 完了 | Minor 2件（下記） |
- 残指摘（Minor 含め全件・重大度つき): ...
- 変更ファイル全一覧: ...
```

## やらないこと

- 承認前の worker 起動
- 書き込み範囲が交わるサブタスクの並列実行
- worker の自己申告だけを根拠にした「完了」報告
- 分解できないタスクの無理な並列化（1本で済むものは fan-out を使わず普通に実行する）

## 連携

- 分解の前提となる仕様が曖昧なとき → 先に `task-brief` で完了条件を確定させる
- 各成果の検証 → `verify-fresh`（このスキルの手順4はその定型呼び出し）
- 作業全体が長時間に及ぶとき → `long-run` の報告規律・停止条件を併用する
- モデルの振り分け原則 → `model-setup/MODEL-GUIDE.md`（worker=sonnet / scanner=haiku）
