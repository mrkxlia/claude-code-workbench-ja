# ナレッジベース (/kb)

複数セッション・複数リポジトリで解決した知見を `~/.claude/knowledge/` に貯め、
次に同じ問題へぶつかったとき即座に引けるようにするスキル。

- **インデックス** `~/.claude/knowledge/index.md` … 1エントリ＝1行。`@import` で全
  セッションに常時読み込まれている（あなたのコンテキストに既に入っているはず）。
- **トピック本体** `~/.claude/knowledge/topics/<topic>.md` … エントリの中身。
  オンデマンドで開く（`git.md` / `docker.md` / `claude-code.md` など kebab-case）。

公式の自動メモリ（プロジェクト単位）の構造をリポジトリ横断に広げたもの。
インデックスは **200行 / 25KB 以内**に保つ（自動メモリと同じ予算）。

このスキルは3モードで動く。

---

## モード1: 記録 — `/kb`（既定）

直近の会話で解決した問題（または有用な発見）を1エントリとして記録する。

1. **重複チェック**: `~/.claude/knowledge/index.md` を読む（多くは既にコンテキスト内）。
   同じ問題が既にあれば、新規追加せず既存エントリを更新する。
2. **トピック選定**: 内容に合う `topics/<topic>.md` を選ぶ（git / docker / python /
   node / claude-code / shell など）。無ければ新規 kebab-case ファイルを作る。
3. **本体を追記**: 下記フォーマットで `topics/<topic>.md` の末尾に追記する。
4. **インデックスに1行**: `index.md` の「エントリ一覧」に
   `- [KB-YYYYMMDD-NN] <タイトル> — topics/<topic>.md #タグ` を追記する。
   採番 `NN` は同日内の連番（その日の既存エントリを見て決める）。
5. **予算チェック**: `index.md` が 200行 / 25KB を超えたら、最も古いエントリ行を
   `~/.claude/knowledge/index-archive.md` へ移す（topics 側の本体は消さない）。

### エントリ・フォーマット（topics/*.md に追記する形）

```
## KB-20260612-01: <一行タイトル>
- 日付 / 出典: <リポジトリ名 or プロジェクト>（session: <id 先頭8桁>）
- 環境: <ツール・バージョン（例: git 2.43 / docker 26 / node 20）>
- 問題: <何が起きたか。他リポジトリでも起こり得る一般形で>
- 原因: <なぜ起きたか>
- 対処: <どう直したか。コピペで再現できる手順・コマンド>
- 物証: <エラーメッセージの核心1行 / 確認に使ったコマンド>
- タグ: #git #docker
- 昇格: <成果物パス>   # 任意。self-improve が rule/skill/CLAUDE.md に昇格させたら記入される
```

> `- 昇格:` 行は**任意**（無くてよい）。`self-improve` の `/improve-apply` が、この知見を
> 恒久成果物（`.claude/rules`・skill・CLAUDE.md など）へ昇格させたときに**一度だけ追記**する。
> 未使用の既存エントリと完全に後方互換。

## self-improve との連携（昇格ライフサイクル）

`knowledge-share` と `self-improve` を両方入れると「**捕捉 → 再発検知 → 恒久成果物へ昇格 →
リンク戻し**」の閉ループになる（片方だけでも単体動作する）。kb 側の関与は次の**追加のみ**:

- **昇格候補マーク**: index 行のタグ領域に `#promote`（昇格候補）/`#promoted`（昇格済み）を付けてよい。
  既存の `#tags` と同じ位置に置く（例: `… topics/git.md #git #promote`）。
- `/kb-harvest` が反復・ワークフロー級の知見に `#promote` を付け、`/improve-scan` に拾わせる。
- 昇格の確定（タグの `#promote→#promoted` 化・`- 昇格:` 追記）は **self-improve 側（improve-apply）**が行う。

> **`/kb search`（モード2）は読み取り専用のまま**で、再利用カウント等の書き込みはしない
> （検索が副作用を持たない＝最小驚き原則）。再発の検知は self-improve がログから行う。

---

## モード2: 検索 — `/kb search <語>`

エラーを再解決する前に、既存の知見を2段で探す。

1. **1段目（インデックス）**: `grep -i "<語>" ~/.claude/knowledge/index.md`
   — タイトル・タグから当たりを付ける。
2. **2段目（本体）**: ヒットした topic、または
   `grep -rin "<語>" ~/.claude/knowledge/topics/` で本体を確認する。
3. 見つかれば、その対処を要約してユーザーに提示する（「過去に KB-… で解決済み」）。
   見つからなければ「既存ナレッジには無し」と伝えてから通常どおり調査する。

> エラーに直面したとき、ユーザーが明示しなくても、まず黙ってこの検索を走らせて
> 既知かどうかを確認してよい（既知ならいきなり解決を提示できる）。

---

## モード3: 昇格 — `/kb promote`

特定プロジェクトに閉じて溜まっている知見を、横断ナレッジへ一般化して引き上げる。

1. 候補元を読む（あるものだけ）:
   - 自動メモリ `~/.claude/projects/<project>/memory/MEMORY.md`
   - プロジェクトの `./CLAUDE.md`
2. その中から **他のリポジトリでも役立つ**ものだけを選ぶ
   （そのプロジェクト固有の規約・ディレクトリ名・ビジネスルールは昇格しない）。
3. プロジェクト固有の語を一般形に書き換えてから、モード1の手順で記録する。
   元の出典は「日付 / 出典」欄にだけ残す。

---

## ハードルール（記録時に必ず守る）

- **機密サニタイズ**: APIキー・トークン・パスワード・内部ホスト名・IP・顧客データ・
  個人情報は記録しない。エラーメッセージは核心の1行だけに切り詰める
  （スタックトレース全文やログの貼り付けはしない）。
- **一般化**: 本文は他リポジトリで再利用できる形にする。リポジトリ固有の情報
  （社内パス・固有名）は「日付 / 出典」欄に隔離し、問題・原因・対処の本文には出さない。
- **小さく保つ**: 1エントリは簡潔に。index.md は予算（200行 / 25KB）を超えさせない。
- **重複させない**: 記録前に必ず index を確認し、既存があれば更新で済ませる。
