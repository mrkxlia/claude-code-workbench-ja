# review-panel — 敵対的パネルレビュー（/review-panel）

コード差分・実装計画・ドキュメントを、**異なるペルソナの複数サブエージェント**にレビュー
させるスキルです。1名の意見ではなく、独立回答（ブラインド）→ 相互批判 → 応答・譲歩 →
統合、という**討論つき**の結論を返します。あなた（メインの Claude）が**ファシリテーター**
を務めます — サブエージェントはサブエージェントを起動できないため、ラウンド進行・匿名化・
ゲート適用はすべてこのスキルの手順に従ってあなたが行います。

パネリストの実行は **`panel-reviewer`** サブエージェント（ペルソナは `personas.md` から
注入）に委譲し、deep モードでは **`panel-verifier`**（引用検証）と **`panel-judge`**
（裁定者）を追加起動します。基本は**外部 CLI・git・ネットワーク不要**で動きます。

## 使い方

- `/review-panel この実装計画をレビューして（…計画本文…）` — **軽量4ラウンド**（既定）
- `/review-panel deep @docs/design.md` — **検証＋裁定つき**（レポートファイル出力）
- `/review-panel codex この差分をレビューして` — **外部パネリスト（Codex）を混成**
- `/review-panel kiro この差分をレビューして` — **外部パネリスト（Kiro）を混成**
- `/review-panel deep codex kiro 5名で <対象>` — 両方混成・人数上書き（上限5名）

「パネルレビューして」「複数の視点で徹底的に」「レビュー会議にかけて」「敵対的にレビュー
して」のような自然文でも発動します（「Codex も混ぜて」「Kiro も混ぜて」でそれぞれの指定と
同じ）。

### 引数の解釈

- `deep`（`--deep`）… 検証フェーズ＋裁定フェーズを追加し、レポートをファイル出力する
- `codex`（`--codex`）… 外部パネリスト（`panel-codex` 経由の Codex CLI）を1名追加する。
  未導入・失敗時は欠席としてパネルは内部のみで続行する
- `kiro`（`--kiro`）… 外部パネリスト（`panel-kiro` 経由の Kiro CLI）を1名追加する。
  未導入・失敗時は欠席としてパネルは内部のみで続行する。**codex と同時指定も可**
  （両方混成 — それぞれ独立に欠席縮退する）
- 残りが**レビュー対象**。`@ファイル` 参照・「この差分」「このプラン」等の文脈参照も可。
  対象が特定できなければ質問して停止する
- 「5名で」等の人数指定があれば従う（上限5名。既定は light=3名 / deep=4名、外部パネリスト
  を2名〔codex+kiro〕とも指定した場合は既定+1名〔light=4名 / deep=5名〕にして内部
  パネリストを最低2名確保する）

## モード

- **light（既定）** — 4ラウンド（R0 実施判断 → R1 ブラインド並列回答 → R2 相互批判 →
  R3 応答・譲歩）＋統合。出力は会話内のみ。Task 呼び出し 6〜9回
- **deep** — light の後に検証（`panel-verifier`・引用の実在照合）と裁定（`panel-judge`・
  対立への判定と最終評決）を追加。レポートを `docs/review-panel/` にファイル出力
- **外部混成（opt-in・codex / kiro）** — どちらのモードにも外部パネリストを混成できる
  （codex・kiro は独立に指定可・同時指定も可）。1名あたりの参加は light=R1 のみ /
  deep=R1＋R3（CLI 呼び出しは1名につき最大2回）

## 他スキルとの違い（重要）

- **`/peer`（ai-peer）** … 1名のセカンドオピニオン（実装前レビュー・壁打ち）。
  「もう1つの視点」で足りるならそちらが軽い。本スキルは**複数・敵対的・討論あり** —
  重要な設計判断・リリース前・意見が割れそうな対象に使う
- **内蔵 `/code-review`・`/codex-review`（codex-bridge）・`/kiro-review`（kiro-bridge）** …
  単独レビュアーによるコードレビュー。行レベルの網羅的な指摘が目的ならそちら。本スキルは
  **指摘を討論でたたき合わせて生き残ったものだけを返す**のが目的
- **`/verify-fresh`（model-setup）** … 成果物と完了条件の突き合わせ検証。合否判定は
  そちら、**多視点の批評と対立の裁定**は本スキル

## ペルソナ編成（既定）

ペルソナの定義本文は同ディレクトリの `personas.md` を読み、選んだブロックを Task
プロンプトに**そのまま注入**する。悪魔の代弁者（③）は反グループシンクの核として**常設**。

| 対象タイプ | light（3名） | deep（4名目に追加） |
|-----------|-------------|--------------------|
| コード差分 | ①正確性ホーク ②セキュリティ監査役 ③悪魔の代弁者 | ⑥コスト・保守モデラー |
| 実装計画・設計 | ⑤実現可能性アナリスト ③悪魔の代弁者 ⑥コスト・保守モデラー | ①正確性ホーク |
| ドキュメント | ④明晰性エディタ ①正確性ホーク ③悪魔の代弁者 | ⑤実現可能性アナリスト |

codex・kiro のいずれか一方のみ指定時は、上記の1枠を該当エージェント（`panel-codex` /
`panel-kiro`）に置き換える（割り当ては ③ か ①。異種モデルの独立視点が最も効く観点に
使う）。**両方指定時は2枠を置き換え、③悪魔の代弁者を `panel-codex`、①正確性ホークを
`panel-kiro`（対象タイプによって表に①が無い場合は逆でもよい。要は2名で③と①を分担する）
に割り当てる**うえで、既定人数を+1（light=4名 / deep=5名）して内部パネリストを最低2名
確保する。人数上書き時も同一ペルソナの重複は不可・上限5名。

## フロー

### Round 0 — 実施判断と編成承認（Task 起動なし）

1. 対象が単純（1関数の是非・単純な質問など）ならパネルは過剰。`/peer` か `/code-review`
   を提案して確認する
2. 実施する場合: 対象タイプを判定し、**ペルソナ編成案・モード・対象パッケージの要約**を
   提示して、**編成の承認を得てから**起動する（承認前にパネリストを起動しない）
3. **対象パッケージを1回だけ構築する** — 差分/プラン本文/抜粋を行番号つきの1ブロックに
   整形（目安400行以内。超える場合は関連箇所に絞るか、リポジトリ内ファイルは path:line
   指定で渡してパネリスト自身に Read させる）。以後の全ラウンドでこの同一パッケージを
   再利用し、都度の再収集をしない

### Round 1 — ブラインド並列回答

- `panel-reviewer` を**1ターンで一斉起動**する（並列上限3。4名以上は 3＋残りの2波）。
  codex・kiro 指定時は同じ波で `panel-codex`・`panel-kiro`（指定された方）も起動する
- 各 Task に渡すもの: ①ペルソナブロック ②ラウンド種別 `R1` ③対象パッケージ
  ④戻り値契約（指摘リスト〔主張1文・根拠・物証 file:line か行引用・重大度 高中低〕＋
  ブラインドスコア 1〜10 ＋最重要指摘1件）
- **他のパネリストの存在・回答は一切渡さない**（ブラインド性は「渡さない」ことで
  構造的に担保する）
- **ゴーストパネリスト検証** — 受領時に「非空・契約準拠（物証とスコアがある）・他回答の
  実質コピーでない」を確認。違反は**1回だけ**同条件で再委譲し、それでも駄目なら
  そのパネリストを「欠席」と記録して先へ進む。**欠席者の声を捏造しない**

### Round 2 — 相互批判（匿名化配布）

- R1 の各回答を**匿名化**する — ペルソナ名・自己言及を剥がし「回答1」「回答2」…として
  指摘内容だけを転記する
- 各パネリストを再起動し、**自分以外の回答**＋対象パッケージ＋契約を渡す。契約:
  「疑わしい」だけの批判は禁止。**「この入力で壊れる」「対象の◯行目と矛盾する」形式の
  具体的批判**か、「同意＋追加根拠」か、「批判なし」のみ
- **具体性ゲート** — 受領時に、反例・矛盾箇所の特定がない批判を破棄する（破棄したものは
  統合時に件数と理由を開示する）
- light+外部混成の場合、外部パネリスト（Codex・Kiro）はこのラウンドに参加しない
  （R1 所見は匿名化されて内部パネリストの批判対象になる）

### Round 3 — 応答・譲歩

- **批判を受けたパネリストだけ**を再起動する（批判ゼロならスキップして Task を節約）。
  渡すもの: 自分の R1 所見＋自分宛の批判一覧＋対象パッケージ
- 契約: 批判1件ごとに**維持（追加根拠必須）／譲歩（理由必須）／修正**＋改訂後の指摘リスト
- **追従的収束の検出** — 理由なき全面譲歩はフラグを立て、その譲歩を無効（元の指摘を
  「対立のまま」扱い）とする
- deep+外部混成の場合、各外部パネリスト宛の批判があればそれぞれ `panel-codex` /
  `panel-kiro` を R3 で再起動する（各自2回目・最後）

### Synthesis — 統合（Task 起動なし）

以下の形式で会話に出力する。**指摘は自己選別せず全件・重大度つき**で載せる（低重大度も
省かない）。対立は**無理に収束させず対立のまま**提示する。

```
## パネルレビュー結果（light・パネリスト3名）
ブラインドスコア分布: 6 / 4 / 7 — 全員一致警告: なし

### 合意した指摘（全件・重大度つき）
| 重大度 | 指摘 | 物証 (file:line) | 支持 |

### 未解決の対立
| 論点 | 一方の主張 | 他方の反論 | 状態 |

### 譲歩履歴 / 破棄した批判（具体性ゲート・件数と理由）

### ファシリテーター私見（パネルの結論とは分離）
```

- **全員一致警告** — スコアの分散が小さく対立もほぼ無い場合は「⚠️ 全員一致（最も危険な
  失敗モード）: 相関バイアスの可能性あり」を明記する（deep なら裁定者が見落とし仮説を出す）
- **ファシリテーター私見の分離** — あなた自身の意見はパネリストへのプロンプトに混ぜず、
  この分離セクションでのみ述べる

### deep 追加フェーズ

**検証（`panel-verifier`）** — R3 を生き残った全指摘の物証（指摘 ID・file:line または
対象内行番号・引用文）だけを渡す（討論ログは渡さない）。戻り値は
`CONFIRMED / LINE-MISMATCH（正行併記）/ NOT-FOUND` の判定表のみ。

**裁定（`panel-judge`）** — R1〜R3 に関与していない fresh context に、①検証結果つき
指摘一覧 ②対立ログ（批判→応答の対）③ブラインドスコア分布 ④対象パッケージを渡す。
戻り値: 対立ごとの裁定（採用/棄却/保留＋根拠）・最終重大度の再判定（NOT-FOUND は棄却か
根拠差し戻し）・全員一致チェック（無風なら見落とし仮説を最低1つ）・最終評決
（承認/条件付き承認/差し戻し）。

**レポート出力** — Synthesis＋検証列＋裁定セクションを `report-template.md` に沿って
`docs/review-panel/<YYYYMMDD>-<slug>.md` に書き出し（書くのはあなた。書き込み先はこの
1ディレクトリのみ）、会話にはサマリとパスを提示する。

## トークン規律

- 既定 light=3名 / deep=4名（外部パネリスト2名同時指定時は+1）、上限5名。
  並列上限3（超える分は2波に分ける）
- 対象パッケージは1回構築・全ラウンド再利用（目安400行）
- R2/R3 の配布は**指摘表のみ**（対象の全文再掲をしない。必要箇所は抜粋）
- 批判ゼロのパネリストの R3 はスキップ。外部 CLI 呼び出しは**各外部パネリストにつき**
  最大2回（light は1回）
- 引用検証は haiku（`panel-verifier`）。判断を伴う役だけ inherit

## このスキルがやらないこと

- レビュー対象や他のファイルを編集する（deep のレポート出力先 `docs/review-panel/` を除く）
- 指摘の修正を実装する（修正が必要なら統合後に別途提案する）
- パネリスト間の複数往復討論（R2→R3 の1往復まで。トークン爆発を防ぐ）
- 欠席パネリストの意見の捏造・対立の無理な収束・私見のパネル結論への混入
- 単純な対象への濫用（Round 0 で `/peer`・`/code-review` へ誘導する）
