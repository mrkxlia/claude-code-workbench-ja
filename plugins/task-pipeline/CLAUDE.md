# CLAUDE.md — タスクパイプラインプロジェクト（サンプル）

## ハードルール（常に適用）

1. **人間承認チェックポイントを飛ばさない** — 要件承認・ブリーフ承認・最終レビューの3点では、明示的な承認があるまで先に進まない
2. **出力ディレクトリの外に書かない** — 成果物ビルダーが作成・編集してよいのは出力ディレクトリだけ。コード・既存資料・設定には触れない（出力ディレクトリ外への書き込みは PreToolUse フックでも確認される）
3. **担当領域を越境しない** — レビュアーは成果物を直さない。ビルダーは要件・ブリーフを書き換えない
4. **要件を推測で埋めない** — 不明点は「未解決の質問」として人間に確認する
5. **機密情報・個人情報・トークンを成果物に書き込まない**
6. **受け入れ基準のセルフチェックをせずに「完了」と言わない**
7. **既存仕様（SPEC.md）に黙って反しない** — `SPEC.md`（成果物仕様）がある場合、その `[確定]` 要件（`D-NN`）を変える/廃止するなら、ブリーフの「既存仕様への影響」で明示し承認を得る。承認された変更は最終承認後に SPEC.md へ増分反映する。原則は **「1 Todo = 1 Commit = 1 Spec Update」**（成果物の区切りごとに仕様も更新して同期させる。`spec-sync-reminder` フックが更新漏れを促す）

## 中間成果物へのポインタ

| ドキュメント | 内容 |
|-------------|------|
| `SPEC.md` | 既存仕様（成果物仕様の spec of record）。`/spec-extract` で逆引きし、最終承認後に増分更新して維持する。researcher / reviewer が一次参照する |
| `docs/task-pipeline/<slug>/` | task-pipeline が生成するタスクごとの中間成果物（進行状況・調査・要件・ブリーフ） |
| `docs/task-pipeline/LEARNINGS.md` | ビルダーの「CLAUDE.md への提案」の蓄積。承認されたものはこのファイルのルール・規約に昇格する |

## 5人の専門エージェント（サブエージェント）

成果物の作成は `/task-pipeline` で以下の連鎖に流す。各エージェントの定義は `.claude/agents/` にある。

| エージェント | 役割 | 書き込み範囲 |
|-------------|------|-------------|
| `source-researcher` | 作る前に素材・規約をマッピングする | なし（読み取り専用） |
| `requirements-writer` | 依頼を受け入れ基準つき要件にする | なし（読み取り専用） |
| `brief-writer` | 要件を作業ブリーフにする | なし（読み取り専用） |
| `deliverable-builder` | 成果物の作成（スキル利用可） | 出力ディレクトリのみ |
| `deliverable-reviewer` | 成果物と要件/ブリーフのギャップ報告 | なし（読み取り専用） |

## 利用可能なスラッシュコマンド

| スラッシュコマンド | 使用する場面 |
|-------------------|-------------|
| `/task-pipeline <依頼の説明>` | 成果物を end-to-end で作成する（5エージェント連鎖 + 3チェックポイント） |
| `/clarify <詰めたい要件>` | 要件・構成を一問ずつ徹底質問で詰める（パイプライン内では Phase 2/3 の writer 起動前に自動で回る） |
| `/spec-extract [対象]` | 既存成果物・規約から「成果物仕様」を SPEC.md に逆引き固定する。**既存 SPEC.md の改訂・更新（要件/構成を途中で変える）にも使う**（整合性の土台＋以降の維持） |
| `/notes` | 成果物作成の判断・逸脱を implementation-notes.md に記録する（パイプライン内ではビルダーが自動記録） |

## 並列実行・調査の網羅性（補足）

- **並列実行グループ** — brief の「並列実行プラン」が独立成果物グループ（出力パスが交わらず共有ファイルを
  書かない）を宣言したときだけ、Phase 4 を「共有先行逐次 → 独立グループ並列 → 依存逐次」で実行する。
  グループ境界の越境はオーケストレーターが守る（フックは出力ディレクトリ外のみ確認）
- **調査の網羅性** — source-researcher は明らかに無関係な素材を除き、関連しうる資料を幅優先で広く読む
  （素材の取りこぼしを避ける）

> **コード変更が必要なときは software-pipeline を使う。** task-pipeline はコード以外の成果物
> （図・ドキュメント・レポート）専用。コードの実装・修正は [software-pipeline](../software-pipeline/) へ。
