# kiro-ask — Kiro に相談する／セカンドオピニオン（/kiro-ask）

設計の是非・代替案・デバッグ方針・トレードオフなど、**自由形式の相談**を **Kiro** に
投げて意見をもらうスキルです。ユーザーは Kiro を直接操作しません。Claude Code が
kiro-cli を**非対話モード**（`--trust-tools=read`）で駆動し、Kiro の回答を要約して
提示します。実際の kiro-cli 実行は **`kiro-advisor` サブエージェント**に委譲し、
冗長な出力をメインの文脈から隔離します。

## 使い方

- `/kiro-ask この設計（〜）は妥当？代替案は？`
- `/kiro-ask なぜこのテストが落ちるか方針を相談したい @path/to/test`

「Kiro に相談して」「kiro の意見も聞いて」のような自然文でも発動します。

## 他スキルとの違い

- **kiro-review** … 差分/コードの**指摘**（重大度 P1–P4）。「悪いところを挙げて」。
- **kiro-ask** … 自由形式の**助言・議論**。「どう設計すべき？」「この方針は妥当？」。
- **相談相手を変えたいとき**（[`ai-peer`](../../../../ai-peer/)・[`codex-bridge`](../../../../codex-bridge/)
  セクション）: 依存を増やしたくない・オフライン/git なし環境なら `/peer`（内部 Claude・
  依存ゼロ）、別 Claude の独立見解なら `/ask-claude`、Codex の意見なら `/codex-ask`。
  kiro-ask は相談相手が **Kiro**（`kiro-cli`）である点が違う。

## 前提

`kiro-cli` が導入・認証済みであること（`kiro-bridge/README.md` の「前提」参照）。
未導入・未認証の場合、サブエージェントが日本語で案内して終了します。

## フロー

1. **相談内容を整理する** — 何を判断したいか、選択肢、制約を明確にする。
2. **明らかに関係するファイル内容を用意する** — 相談に必要な抜粋を渡し切る
   （大きすぎる場合は関連箇所に絞る）。
3. **`kiro-advisor` を起動する**（Task ツール） — 相談内容と同梱ファイルを渡す。
   サブエージェントが Kiro を read-only で実行し、回答要約を返す。
4. **結果を提示する** — 結論・根拠・推奨アクション・留意点を見せる。
5. **必要なら橋渡しする** — 方針が固まったら `/kiro-review`（レビュー）や、実装が
   必要なら Claude 自身の実装（kiro-bridge に implement 系はない）に続けることを提案する。

## このスキルがやらないこと

- ユーザーに kiro-cli を手動で叩かせる（すべて Claude が非対話で駆動する）
- リポジトリのファイルを編集する（read-only。kiro-bridge に implement 系スキルはない）
