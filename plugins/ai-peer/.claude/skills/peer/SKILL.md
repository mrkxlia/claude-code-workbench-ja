# peer — 内部ピアに相談する（/peer）

設計・方針・行き詰まりについて、**第二の視点**を内部の Claude サブエージェントから引き出すスキルです。
**外部 CLI・git・ネットワークを一切使いません**（依存ゼロ＝ロックダウン/オフライン/git なし環境でも
確実に動く）。実際の相談は **`peer-engineer` サブエージェント**に委譲し、メインとは別の
fresh context で独立した視点を返させます。

## 使い方

- `/peer plan この実装方針をレビューして（…概要…）` — **実装前のプランレビュー**
- `/peer brainstorm この問題の打開案を一緒に考えて` — **ブレスト/壁打ち**
- `/peer この設計の第二意見が欲しい` — モード省略時は**汎用セカンドオピニオン**

「ピアに相談して」「壁打ちして」「別の視点で」のような自然文でも発動します。

## モード

- **plan（実装前のプランレビュー）** — 設計の穴・暗黙の前提・見落とした代替案・リスクを洗い出す。
  実装に着手する前の「設計の番人」。
- **brainstorm（発想支援）** — 詰まったとき・選択肢が見えないときに案を広げる。
- **review（軽い方針レビュー）** — 変更の方向性が妥当かの相談。**行レベルのコードレビューは行わない**。

## 他スキルとの違い（重要）

- **コードレビューそのもの（バグ/重大度つき指摘）は peer の仕事ではない** → 内蔵 `/code-review`
  （依存なし）や `/codex-review`（Codex CLI）に委譲する。peer は **実装前**と**発想**に軸足を置く。
- **`ask-claude`** … 別の Claude を CLI で起動して独立見解を得る（`claude` CLI が必要）。
  peer は**外部依存ゼロ**で動く点が違う。
- **`/codex-ask`（codex-bridge）** … Codex に相談（`codex` CLI が必要）。

依存を増やしたくない・git なし・オフライン環境なら **peer** を選ぶ。別エンジンの視点が欲しいなら
`ask-claude`（別 Claude）や `/codex-ask`（Codex）を使う。

## フロー

1. **モードと題材を決める** — 引数のモード（plan/brainstorm/review）に従う。無ければ題材から判断し、
   不明なら汎用セカンドオピニオンとして扱う。
2. **レビュー対象を渡し切る** — プラン本文・変更概要・関連コードの**内容そのもの**を
   サブエージェントに同梱する（`peer-engineer` は外部 CLI も diff 取得手段も持たないので、
   **呼び出し側が必要な文脈を渡し切る**）。大きすぎる場合は関連箇所に絞る。
3. **`peer-engineer` を起動する**（Task ツール） — モード・題材・同梱文脈を渡す。
   サブエージェントが独立視点で評価し、要約を返す。
4. **結果を提示する** — 結論・根拠・推奨アクション・留意点を見せる。
5. **必要なら橋渡しする** — 方針が固まったら実装へ、コードレビューが要るなら `/code-review` /
   `/codex-review`、別エンジンの意見が欲しいなら `ask-claude` / `/codex-ask` を提案する。

## このスキルがやらないこと

- 外部 CLI・git・ネットワークに依存する（依存ゼロが売り。別エンジンが要るなら ask-claude / codex-bridge）
- 行レベルのコードレビュー（/code-review・/codex-review に委譲）
- リポジトリのファイルを編集する（相談のみ。実装は別途）
